# Pattern: Visual Dashboard Cards

Dynamic PNG image cards generated by serverless endpoints, embedded inline in agent output for dense at-a-glance information.

---

## The Problem

Text output is information-dense but slow to scan. A reader glancing at a long Slack post has to read through to extract the key numbers. When an operator is processing 30+ agent outputs per day across multiple channels, reading everything in full is not viable.

Slack supports inline images. A well-designed image card can convey at a glance what a paragraph conveys on careful reading: current score, trend, severity, category breakdown. Dashboard cards let the operator skim and dive deeper only when something warrants attention.

---

## The Pattern

Agents construct URLs to serverless image-generation endpoints, passing live data as URL parameters. Slack auto-unfurls the image URL and renders it inline. The card is generated on request (not cached stale), visually consistent with the fleet's design standard, and immediately readable.

No agent stores images. No agent renders images. Agents only construct URLs. The rendering is entirely serverless and stateless.

---

## The Stack

**Generation:** Satori (React JSX → SVG) + Sharp (SVG → PNG). Runs on a serverless function (Vercel, Cloudflare Workers, or equivalent).

**Why Satori:** It takes JSX and produces deterministic SVG without a browser. Fast, cacheable, no headless Chrome overhead. Perfect for programmatic card rendering.

**Why PNG output (not SVG):** Slack's unfurler handles PNG better than SVG in preview rendering. The serverless endpoint rasterizes the Satori SVG into PNG via Sharp before returning.

**Why URL-parameter data:** Agents don't need HTTP POST capability to pass structured data. A GET with query parameters is simple enough for any agent to construct in a single line of code, and the URL itself is the data — no state maintained on the server side.

---

## Anatomy of a Card Endpoint

Example endpoint: `https://[deployment].vercel.app/api/score-card`

Accepts query parameters:
- `title` — card title
- `score` — numeric value 0–100
- `trend` — "up" / "down" / "flat"
- `delta` — numeric change (for display)
- `severity` — "critical" / "high" / "medium" / "low"
- `timestamp` — ISO timestamp

Returns: PNG image at a fixed resolution (typical: 1200x630 for Slack unfurl, 600x400 for inline).

Agent-side usage:

```
https://[deployment].vercel.app/api/score-card?title=Market%20Health&score=72&trend=up&delta=5&severity=medium&timestamp=2026-04-19T14:30:00Z
```

Agent posts the URL in Slack. Slack unfurls. Card renders.

## Typical Card Types in a Fleet

| Card Type | Endpoint | Used By |
|-----------|----------|---------|
| Score gauge | `/api/score-gauge` | Health scores, composite indices |
| Market snapshot | `/api/market-snapshot` | Market monitor, market pulse |
| Alert card | `/api/alert-card` | Critical escalation routing |
| Status overview | `/api/status-overview` | Fleet watchdog, auto-repair |
| Pipeline funnel | `/api/pipeline-funnel` | Opportunity-tracking and workflow agents |
| Severity matrix | `/api/severity-matrix` | Regulatory oracle, compliance hub |

Each card endpoint is ~100 lines of Satori JSX. The whole card system for a fleet typically fits in one serverless deployment with a dozen endpoints.

## Design Standard

Card visual consistency matters for operator pattern recognition. A fleet's cards should share:

**Theme:**
- Dark background (e.g., #0a0a0a or #111) for readability on both light and dark Slack themes
- Single accent color for primary data (fleet brand color)
- Neutral grays for supporting information
- Severity-based color coding: critical → red, high → amber, medium → yellow, low → green

**Layout:**
- Consistent safe zones (content lives within middle 80% of the image)
- Primary number at the largest size — the thing the reader scans first
- Trend indicator immediately adjacent to primary number
- Secondary details in a footer row
- Timestamp in the corner

**Typography:**
- One sans-serif family (Inter, IBM Plex, similar) used consistently
- Weight hierarchy: 800 for primary number, 500 for labels, 400 for footer
- No decorative fonts

Consistent design lets the operator learn the pattern once and read any card instantly.

## Agent Integration

Adding card output to an agent requires exactly one line change in the agent's output format:

```
## Snapshot
![Market Snapshot](https://[deployment].vercel.app/api/market-snapshot?btc=62400&eth=3200&cap=2.4T&trend=up&delta=2.1)
```

Slack renders the image inline. The agent's URL construction step is:
- Gather data (happens anyway in Step 1)
- URL-encode values
- Construct the URL string with query parameters
- Include in the output

No new capability required beyond what the agent already does to format text output.

## Cost and Performance

**Generation latency:** 100–300ms per card with warm serverless function. Not on the critical path of agent execution since the agent just posts a URL.

**Cost:** Vercel hobby tier typically covers thousands of card renders per month for free. A card-heavy fleet might need the paid tier.

**Caching:** Cards are generated on demand with live data — caching is generally undesirable. A timestamp parameter makes each URL unique, bypassing caches.

---

## Why Cards Beat Text for Some Outputs

- **Numbers scan faster than prose.** A "72/100 ▲ +5" card communicates state in under a second. The equivalent prose requires reading a sentence.
- **Severity color is pre-attentive.** The brain processes color-coded risk before word-level comprehension. A red card flags critical items without the reader needing to read.
- **Trend arrows are universal.** Up/down/flat arrows are understood across language and context. They reduce translation overhead.
- **Consistent layout trains pattern matching.** Operators see 10+ cards a day in the same format — they pattern-match the state before reading specifics.

---

## When NOT to Use Cards

- **For dense tabular data.** A 10-row table is often better as a markdown table than as an image. Images don't copy/paste as text.
- **For linked content.** Cards are images. They can't have clickable internal links. If the output is a list of items each needing navigation, use text.
- **When precision matters.** Images lose precise alignment that a markdown table preserves. A reconciliation report belongs in markdown.
- **When the audience uses screen readers.** Cards are images; accessibility requires alt text. Always include a text summary alongside critical cards.

---

## Anti-Patterns to Avoid

**Decorative cards.** A card with no numeric content is a wallpaper. If the image isn't conveying data, don't post it.

**Cards that duplicate text.** If the card renders the same three numbers the output text already shows, pick one. Two copies of the same information dilutes the scan.

**Inconsistent design across agents.** If one agent's cards look nothing like another's, the operator loses pattern-recognition benefit. Enforce a shared design standard.

**Cards too complex.** A card with 12 numbers becomes harder to scan than the text. Limit cards to 1 primary number + 2–3 supporting values + optional trend.
